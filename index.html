js深入理解(4)-变量、作用域和内存机制(draft)

Posted on 2018-09-15
Words count in article: 1,998 字 | Reading time ≈ 7 分钟

变量：
ECMAScript变量包含两种类型的值：基本类型和引用类型。

引用类型的值是保存在内存中的对象，操作对象时，实际上是在操作对象的引用而不是实际的对象。 故而，引用类型的值是按引用访问的。

变量当做参数传递：
js所有参数都是按值传递的。

向函数参数传递基本类型的值的时候，被传递的参数会被复制给一个局部变量（即命名参数，就是arguments对象的第一个元素。）

function add(num){
num += 10;
retunrn num
};
var count = 12;
var result = add(count);
console.log(count) // 20; 没有变化
console.log(result) // 30;

而向参数传递引用类型的值时，会把这个值在内存中的地址复制给局部变量。 故而局部变量的变化会反应在函数外部。

function setName(obj){
obj.name = ‘harry’;
}
var person = {};
setName(person);
console.log(person.name); // harry

变量被传递给setName函数之后被复制给了obj，obj和person引用的是同一个对象。

之所以是按值传递而非按引用传递，完全取决于被对象被重写之后的函数返回：

function setName(obj){
obj.name = ‘harry’;
var obj = new Object();
obj.name = ‘potter’
}
var person = {};
setName(person);
console.log(person.name); // harry

此处parson.name 仍然为harry而非改写之后的potter， 说明即使在函数内部修改了参数的值，原始的引用仍然保持未变。

简单概括：
var a = {}; // 此处 变量a保存指向{}的指针。
var b = a; // 此时， b===a， b也保存指向{}的指针。
a.name = ‘harry’; // 当对象添加属性，b因为指向相同的指针，故b也有相应的属性。
b // {name: ‘harry’};

var a = {}; // 此时a重写，重新保存另一个对象的指针，自此，a变量断开与原先对象的指向关系。
b // {name: ‘harry’}; 自此a的修改再无法影响到b。

以上在js系列1引用章节有提及。

内存：

不管什么程序语言，内存生命周期基本是一致的：

  1. 分配你所需要的内存

  2. 使用分配到的内存（读, 写）

  3. 不需要时将其释放/归还。

    JS 中的内存回收
    垃圾回收算法主要依赖于引用的概念. 在内存管理的环境中, 一个对象如果有访问另一个对象的权限（隐式或者显式）, 叫做一个对象引用另一个对象. 例如: 一个Javascript对象具有对它原型的引用（隐式引用）和对它属性对象的引用（显式引用）.

    引用计数
    跟踪记录每个值被引用的次数，声明一个变量并赋其一个引用类型的变量时，该次数就是一。 如果同一个值又被赋予另一个变量，该次数加1。 相反，包含当前引用的变量被改写获得另一个值，该次数-1，当次数为0时，说明没有办法再访问当前值，该值会在下次垃圾收集器运行时被收集。
    var a = {}; // 此时，次数为1；
    var b = a; // 次数+1；

    var a = []; // a被改写，次数-1；
    var b = a; // b也被改写，自此次数减至0；之后{}会被回收。

    限制: 循环引用
    function f() {
    var o1 = {};
    var o2 = {};
    o1.p = o2; // o1 引用 o2
    o2.p = o1; // o2 引用 o1. 这里会形成一个循环引用
    }
    f();

实际例子:

var div;
window.onload = function(){
div = document.getElementById(“myDivElement”);
div.circularReference = div;
div.lotsOfData = new Array(10000).join(“*”);
};
上面的例子里, myDivElement 这个 DOM 元素里的 circularReference 属性引用了 myDivElement, 造成了循环引用. IE 6, 7 使用引用计数方式对 DOM 对象进行垃圾回收. 该方式常常造成对象被循环引用时内存发生泄漏. 现代浏览器通过使用标记-清除内存回收算法, 来解决这一问题.

标记-清除算法

自动垃圾收集机制（GC：Garbage Collecation）

这个算法假定设置一个叫做根 root 的对象（在Javascript里，根是全局对象）. 定期的, 垃圾回收器将从根开始, 找所有从根开始引用的对象, 然后找这些对象引用的对象, 从根开始,垃圾回收器将找到所有可以获得的对象和所有不能获得的对象.

标记：标记的过程其实就是，遍历所有的GC Roots，然后将所有GC Roots可达的对象标记为存活的对象。

清除：清除的过程将遍历堆中所有的对象，将没有标记的对象全部清除掉。

当程序运行期间，若可以使用的内存被耗尽的时候，GC线程就会被触发并将程序暂停，随后将依旧存活的对象标记一遍，最终再将堆中所有没被标记的对象全部清除掉，接下来便让程序恢复运行。

内存泄露
1.意外创建全局变量
2 未销毁的定时器和回调函数
var serverData = loadData();
setInterval(function() {
var renderer = document.getElementById(‘renderer’);
if(renderer) {
renderer.innerHTML = JSON.stringify(serverData);
}
}, 5000); // 每 5 秒调用一次
如果后续 renderer 元素被移除, 整个定时器实际上没有任何作用. 但如果你没有回收定时器, 整个定时器依然有效, 不但定时器无法被内存回收, 定时器函数中的依赖也无法回收. 在这个案例中的 serverData 也无法被回收.

闭包

DOM 引用
对于一个 Dom 树的叶子节点的引用. 举个例子: 如果我们引用了一个表格中的 td 元素, 一旦在 Dom 中删除了整个表格, 我们直观的觉得内存回收应该回收除了被引用的 td 外的其他元素. 但是事实上, 这个 td 元素是整个表格的一个子元素, 并保留对于其父元素的引用. 这就会导致对于整个表格, 都无法进行内存回收. 所以我们要小心处理对于 Dom 元素的引用.

执行环境与作用域:
执行环境定义了变量或者函数有权访问其他数据
每个执行环境都一个对应的变量对象，比如在web浏览器中，全局环境被认为是window对象。
某环境代码执行完成，环境被销毁。其变量与函数定义也被销毁。

所有变量（包括基本类型和引用类型）都存在于一个执行环境（也称为作用域）当中，这个执行环境决定了变量的生命周期，以及哪一部分代码可以访问其中的变量。以下是关于执行环境的几点总结：

执行环境有全局执行环境（也称为全局环境）和函数执行环境之分；
每次进入一个新执行环境，都会创建一个用于搜索变量和函数的作用域链；
函数的局部环境不仅有权访问函数作用域中的变量，而且有权访问其包含（父）环境，乃至全局环境；
全局环境只能访问在全局环境中定义的变量和函数，而不能访问局部环境中的任何数据；
变量的执行环境有助于确定何时释放内存。
延长作用域链：try-catch语句的catch块；with语句。
JavaScript没有块级作用域：for语句创建的变量i即使在for循环执行结束后，也依旧会存在于循环外部的执行环境中；使用var声明会自动被添加到最接近的环境中。

当代码在一个环境中执行的时候，就会形成一种叫做作用域链的东西，它的用途是保证对执行环境中有访问权限的变量和函数进行有序访问（指按照规则层次来访问），作用域链的前端，就是执行环境的变量对象。
